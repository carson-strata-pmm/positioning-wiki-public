# Value over Features

## Definition

Value over features is the principle that B2B messaging must be organized around what the buyer *gets* (outcomes, transformation, solved problems), not what the product *does* (capabilities, features, technical specifications). Features describe the product's mechanics. Value describes the buyer's changed world. The buyer doesn't care how the engine works -- they care where the car takes them.

## How It Works

### The Distinction: Capabilities vs. Value

| Level | What It Answers | Example (TeamOhana) | Where It Lives |
|-------|----------------|---------------------|----------------|
| **Feature** | "What does the product do?" | Real-time data sync, approval workflows, automated reconciliation | USPs (Layer 6) |
| **Capability** | "What does this enable?" | Cross-functional headcount visibility, governance controls | Value Pillars (Layer 5) |
| **Value** | "What does the buyer get?" | Strategic alignment, built-in compliance, faster decisions | Outcomes (Layer 7) |
| **Problem** | "What pain goes away?" | Untrusted data, lack of control, wasted time | Pain Pillars (Layer 3) |

Feature-led messaging fails because it requires the buyer to do the translation work: "OK, so it has real-time data sync... what does that mean for me?" Value-led messaging does the translation for them: "Your headcount data is finally trustworthy, so you can make decisions in real-time instead of chasing spreadsheets."

### How Value Pillars Are Constructed: The Inversion Pattern

Strata builds value pillars by inverting pain pillars. Every pain pillar describes a broken state; the corresponding value pillar describes the solved state. This structural mirror ensures that every value claim is directly connected to a problem the buyer recognizes.

**Three inversion types:**

| Type | Pattern | Example | When to Use |
|------|---------|---------|-------------|
| **Negation** | Directly negate the pain | "Untrusted data" → "Real-time visibility" | Cleanest and most direct. Default choice. |
| **Aspiration** | Name the aspirational state beyond the pain | "Staffing challenges" → "Scale your front office" | More emotionally compelling. Good for owner-operators. |
| **Mechanism** | Name *how* the pain is solved | "Terrible CX" → "Single point of contact" | When the mechanism itself is a differentiator. |

### The Value Hierarchy: Pillar → USP → Outcome

Each value pillar contains two sub-layers:

```
Value Pillar (what the buyer gets -- 3-5 words)
    |
    USPs (specific capabilities that deliver the value -- 2-4 per pillar)
    |
    Outcomes (measurable business results -- 1-2 per pillar)
```

**Avoca's "Book more jobs" pillar:**
- **Value pillar:** Book more jobs
- **USPs:** AI answers every call in <3 seconds, automated lead follow-up, outbound nurture campaigns
- **Outcome:** "Never miss a call. Full board year-round."

Features (USPs) sit *under* value, not above it. The value pillar is the headline; the features are the supporting evidence. This structure ensures that marketing leads with value and proves with features, rather than leading with features and hoping the buyer figures out the value.

## Examples from Client Work

### Strong Value-over-Features: Avoca

Avoca's three value pillars are pure aspiration: "Book more jobs," "Deliver exceptional experiences," "Scale your front office." None of these mention AI, agents, or technology. They describe what the owner-operator *gets*. The features (AI call handling, human-in-the-loop escalation, <3 second answer time) sit below as proof. An owner-operator reads "Book more jobs" and immediately cares -- they don't need to understand the technical mechanism first.

### Strong Value-over-Features: Agency

Agency's value pillars are bold and memorable: "Single point of contact," "Complete context on every customer," "Infinite capacity to serve." These are aspirational mechanism statements -- they describe a new operational reality, not a product feature. "Infinite capacity to serve" is a fundamentally different promise than "AI-powered customer support automation."

### Weak Value-over-Features: BrightEdge

BrightEdge's value pillars are: "Work automation," "Insights," "Visibility." These are functional labels, not value statements. "Insights" could describe any analytics product. Compare with TeamOhana's "Real-time visibility" -- it names the specific *kind* of visibility and implies the pain it resolves (data was previously delayed or untrustworthy). BrightEdge's pillars describe capabilities, not transformation.

### Weak Value-over-Features: Snappy

Snappy's value pillars are: "Expertise," "Scale," "Delight." These are single-word abstractions. "Expertise" doesn't tell the buyer what they get -- expert at what? For whom? Compare with Avoca's "Book more jobs" or Unity Games' "Develop without compromise" -- each tells a mini story in 3-4 words. Single-word pillars force the buyer to guess at the value.

## Patterns & Lessons

- **Features are proof, not headlines.** Lead with value ("Scale your front office"), support with features ("AI handles calls 24/7, human-in-the-loop for complex issues"). The homepage rubric scores "Product Explanation" on whether the buyer understands what they *get*, not what the product *does*.
- **The inversion pattern guarantees problem-value alignment.** If every value pillar inverts a pain pillar, the narrative is structurally guaranteed to address the buyer's actual pain. Value claims that can't be traced back to a named pain are orphaned -- they sound nice but don't connect.
- **Consistent inversion type creates rhythm.** TeamOhana uses negation for all three pillars. Avoca uses aspiration for all three. The consistency creates a pattern the buyer can follow. Mixing types (Agency) works but feels less unified.
- **Value pillars should be 3-5 words, not single words.** "Real-time visibility" > "Visibility." "Book more jobs" > "Revenue." "Deploy without friction" > "Deployment." The extra words add specificity and narrative.
- **Value language should be testable.** "Book more jobs" is a testable claim -- you can measure whether Avoca customers book more jobs. "Empower your team" is not testable. Value pillars that can't be measured can't be proved.
- **The copy bank translates value into formats.** Value pillars feed the copy bank (taglines, 25-word, 50-word, elevator pitch). If the value pillars are abstract, the copy bank will be too. If they're concrete, the copy practically writes itself.
